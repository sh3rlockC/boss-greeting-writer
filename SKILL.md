# Boss 首条打招呼话术

目标：先判断岗位真正要招什么人，再从候选人素材里挑最贴 JD 的点，生成更容易拿到回复的 Boss 首条私聊话术。

## 1. 适用范围

只处理：
- Boss 直聘单个岗位的首条打招呼

不处理：
- 二次跟进话术
- 非 Boss 平台私信
- 完整简历改写
- 面试问答

## 2. 输入

支持输入：
- JD 文本
- JD 截图
- 候选人结构化经历要点
- 候选人长文本素材
- 候选人截图
- 目标公司 / 条线信号：如字节、飞书、阿里、解决方案、客户成功、携程
- 目标风格：自然 / 稳妥 / 大厂HR向
- 补充约束：如“更短”“更保守”“不要强调 AI”“直接给最终版”“进入连打模式”

说明：
- `自然` 和 `稳妥` 是版本偏好
- `大厂HR向` 是表达路由信号，不是第四个输出版本
- 如果用户同时给了公司信号和风格要求，优先保留用户显式要求，再用公司信号微调措辞

输入优先级：
- 文本优先于截图 OCR
- 结构化要点优先于长文本抽取
- 用户指定风格优先于自动推荐

## 3. 输出

默认完整输出：
- 岗位意图判断
- 建议主打卖点
- 不建议强调的内容
- 3 版首条话术
- 推荐使用建议

固定的 3 版首条话术始终是：
- 自然版
- 稳妥版
- 超短首条版

如果用户要求 `大厂HR向`：
- 不新增第四版
- 而是让三版都更短、更规整、更适合扫读

如果用户明确说了以下任一信号：
- `直接给最终版`
- `连打模式`
- `不要解释`
- `不要分析`
- `不要再说给我一版`

则切换到 `直出模式`：
- 只输出 1 条最终可发送话术
- 不再输出岗位分析、卖点分析、版本对比和推荐说明
- 语气以“复制即可发送”为目标

## 4. 工作流程

按以下顺序执行：
1. 解析输入来源。
2. 如果输入包含截图，先调用 `paddleocr-text-recognition` 提取文本。
3. 如果候选人素材是长文本，先抽取项目、场景、职责锚点和可用的身份标签。
4. 判断岗位核心招聘意图。
5. 如果 JD 或用户要求带有明显公司 / 条线信号，再判断招聘者更吃哪种表达。
6. 先按 `岗位贴合度 > 可证实性 > 扫读友好度` 给候选人素材排序。
7. 只筛选最贴 JD 的 2 到 3 个经历点。
8. 对项目名和职责名做风险降级，优先保留具体场景而不是放大成泛化 title。
9. 输出自然版、稳妥版、超短首条版。
10. 推荐最适合发送的一版并说明原因。

判断岗位时，先回答一句：

“这个岗位真正要招的人，最核心是来干什么的？”

招聘者表达口味重点看两类信号：
- 公司 / 业务线信号：如字节 / 飞书更偏短、直接、好扫读
- 职能导向信号：如阿里解决方案 / 客户成功更偏场景、方案、落地、协同

如果 JD 明确出现以下信号：
- `AI Native`
- `Agentic Platform`
- `hands-on`
- `MVP`
- `AI coding`
- `Vibe Coding`

优先判断为：
- 偏 `AI Native 产品 / Agent 场景定义 / 原型极速落地`
- 不只是传统流程型 PM，也不是纯策略型产品
- 需要把 `用户痛点判断 + 场景设计 + 亲手跑通原型 + 工程协同调优` 一起写出来

如果 OCR 结果缺词、错行、关键信息不足：
- 明确提示信息不足
- 建议补文本
- 只在必要时给保守版，不要硬编
- 如果 OCR 或图片识别失败，但职位标题、核心关键词、职责主轴仍然清晰可见，可以基于当前可见信息输出保守版首条，不补不可见细节，不假装已完整识别

## 5. 硬规则

必须遵守：
- 目标是拿到回复，不是写得漂亮
- 首条默认 90 到 120 字优先
- 超过 140 字视为偏长，除非用户明确要求更完整
- 优先 2 句话，最多 3 句话
- 第一条消息只做三件事：为什么关注、哪里匹配、方便发简历吗
- 先判断岗位意图，再挑素材，不要反过来
- 每条只打 1 个主亮点
- 最多再带 1 到 2 个辅助点
- 候选素材先按 `岗位贴合度 > 可证实性 > 扫读友好度` 排序
- 每条至少有 1 个真实经历锚点
- 少用“非常热爱、十分关注、期待加入”等虚词
- 默认避免 `贵司`、`有幸`、`真诚自荐`、`衷心期待` 这类模板味表达
- 默认语气自然、克制、像做过事的人
- 不确定时宁可保守，不夸大项目
- “主导”“负责人”“平台”等词要按真实经历降级使用
- 如果项目可以用更具体的场景表述，就不要先上升到“平台”级表述
- 少用 title 自证，多用场景、动作、结果来证明匹配
- 候选人的兴趣、长期关注、重度用户身份只能作为辅助信号，不能压过真实项目经历与职责证据，除非 JD 明确高度依赖该类身份
- 弱匹配时优先写“比较贴”“相近方向”“有持续关注和实践”，不要写成“高度匹配”“丰富经验”“正是你们要找的人”
- 默认少用“完整闭环”“能力匹配”“相关业务探索”“持续调优”“完整链路”“产品化能力”这类泛化 AI/汇报腔表述，优先写清具体做过什么、跑通过什么、解决了什么问题
- 如果 JD 强调 `hands-on`、`MVP`、`跑通原型`、`AI coding`，优先写自己亲手做过什么，不要先写泛化策略话术
- 如果 JD 把行业背景写成加分项而不是硬门槛，没有直接经历时不要硬蹭，只能降级为辅助理解或学习迁移

## 6. 风格原则

自然：
- 更像真人私聊
- 语气稍松，但不散

稳妥：
- 信息规整
- 适合 HR 快速扫读

大厂HR向：
- 这是表达路由信号，不是第四个输出版本
- 更压缩
- 更清楚
- 更强调结构和匹配点
- 适合字节 / 飞书 / 腾讯这类更看重扫读效率的场景
- 推荐通常优先落在稳妥版或超短首条版

如果用户未指定风格，自动推荐，但仍输出三版供选择。

## 7. 公司 / 条线口味路由

字节 / 飞书 / 腾讯 / 月之暗面这类快节奏 AI 公司 HR 扫读场景：
- 更短、更直给、少虚词
- 先说做过什么，再说为什么匹配
- 少讲概念判断，多给具体职责锚点
- 少用“完整闭环”“匹配度较高”等汇报腔

阿里系或解决方案 / 客户成功场景：
- 更强调业务场景、方案设计、推进落地
- 少把自己写成纯产品 title，更多写成懂场景、能协同、能落地的人
- AI 只写到业务结合和方案落地，不堆模型概念

阿里系 `AI Native / Agentic Platform / hands-on` 场景：
- 主亮点优先给 `亲手搭过的 Agent 工作流 / MVP / 原型`
- 辅助点优先给 `用户痛点判断`、`工程协同调优`、`结果验证`
- 如果候选人还有 GTM、上市、销售支持经历，默认后置，不要抢主亮点
- 如果 JD 只把电商或行业背景写成加分项，没有直接经历时不要冒充行业专家

携程这类传统大厂 HR / 招聘经理场景：
- 更稳妥、更规整、更像成熟候选人
- 更重产品运营、需求分析、结果负责、跨团队协同
- 少一点极客味和 AI 发烧友表达

如果没有明确公司信号：
- 以岗位核心招聘意图优先
- 再用默认三种风格去拉开差异

## 8. 输出格式

完整模式下，按以下顺序输出：
1. 岗位意图判断
2. 建议主打卖点
3. 不建议强调的内容
4. 自然版
5. 稳妥版
6. 超短首条版
7. 推荐使用建议

推荐使用建议里必须说明：
- 推荐发哪版
- 为什么推荐
- 适用场景
- 如果用了公司 / 条线路由或 `大厂HR向`，要说明它如何影响了推荐

直出模式下：
- 只输出 1 条最终版
- 默认写成 Boss 上可直接复制发送的口吻
- 不补分析，不写“我给你一版”“推荐版如下”之类前置句

## 9. 降级规则

以下情况要主动保守，不要强写：
- OCR 不完整：明确说明缺口，建议补文本
- JD 很短或很虚：只做保守判断，不做细粒度职责推断
- 候选人素材不足：先指出证据不足，再给低风险版本
- 匹配度较弱：可以表达理解和关注，但不能伪造“做过类似项目”
- 岗位重心判断不清：优先抓最强主轴，其余只做辅助
- 高风险措辞：将“主导”“负责人”“平台”等词降级到真实可证实的表述
- 同一信息有多种来源：优先采用更明确、信息损耗更小的来源
- 用户明确要求“不要强调 AI”：即使 JD 带 AI，也只保留和岗位强相关的业务落地表述，不堆模型概念
- JD 把某行业写成加分项：若候选人无直接经历，不要为了显得更匹配而伪造该行业经验
- `平台` 只有在存在明显平台化能力时才使用，如统一编排、复用配置、多任务或多用户；否则优先改写为具体场景下的工作流、原型或 MVP
- `MVP` 可以用，但最好和具体场景绑定，例如“面向汽车口碑研究的 AI Agent 工作流 MVP”，不要空写成“工作流平台 MVP”

## 10. 典型岗位判断参考

- AI 产品经理：优先看 AI 项目、技术理解、落地经验
- AI Native / Agentic Platform：优先看用户痛点判断、场景定义、hands-on 原型、工程调优、结果验证
- AI 解决方案 / 客户成功：优先看场景理解、方案推进、跨团队协同
- 内容产品 / 用户产品：优先看用户理解、内容场景、重度用户身份
- 数据产品 / 平台产品：优先看数据链路、平台协作、表达规整

## 11. 常见失误

- 看到 JD 里有 AI 就把所有话术都写成 AI 模板
- 把第一条写成长篇自我介绍
- 把所有经历都塞进去，导致 HR 扫不到主亮点
- 为了显得强而滥用“主导”“负责人”等高风险词
- 明明是场景型工作流，却硬写成“平台 MVP”
- JD 明明在看 `hands-on` 和 `MVP`，却先写成泛化的“策略/规划能力”
- JD 只把行业背景写成加分项，却硬把自己写成该行业深度从业者
- 输出三版话术，却不告诉用户推荐发哪版和为什么

压力场景与校验样例见 `references/validation-cases.md`。
